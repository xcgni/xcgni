# Excogni Android: production app plan

Prepared against v0.59.1. Grounded in what the repo already contains, not generic advice.

**BUILD STATUS (v0.60.0): the server/web half of Phase 1 is BUILT.** Done: notification prefs
(migration 0033 + Settings "App reminders" section, all opt-in), GET /api/notify/plan (reminder +
capped conditionals with the user's timezone), native.ts bridges (syncNotificationSchedule,
registerBackButton, keepScreenAwake via the Wake Lock API - no plugin needed, isNative exported,
hideSplash finally called), layout native init on mount, wake lock during reaction
calibration/trials, /.well-known/assetlinks.json route (serves only when env ANDROID_CERT_SHA256
is set), capacitor-shell/error.html offline page + errorPath in the config and generator, deps
added (@capacitor/local-notifications, @capacitor/app). Auth stays magic-link email - the SAME
account as the web; the phone's Google account is NOT the identity (Google Sign-In is a possible
later convenience, not the login key).

**YOUR-MACHINE STEPS (the native half, in order):**
1. npm install (new deps), commit package-lock.json.
2. node scripts/make-variant.mjs serious (regenerates config with xcgni.com + errorPath).
3. npx cap sync android.
4. Android Studio: in AndroidManifest.xml add the App Links intent-filter for
   https://xcgni.com (android:autoVerify="true", scheme https, host xcgni.com) on MainActivity.
5. Generate the upload keystore, get its SHA-256 (keytool -list -v), set ANDROID_CERT_SHA256 on
   the server (compose env), redeploy, verify https://xcgni.com/.well-known/assetlinks.json.
6. Build a debug APK to your phone; run the acceptance loop: install -> request magic link ->
   tap it in your mail app -> lands logged-in IN the app -> enable the reminder in Settings ->
   background the app -> reminder fires at the chosen time -> tap lands in practice.
7. Add a small notification icon (ic_stat_icon) in android/app/src/main/res (white-on-transparent).

**The one-line architecture, confirmed:** a Capacitor remote-URL shell around the live site, same
backend, same Postgres, zero duplicated logic. This is already the chosen design in the repo
(capacitor.config.ts points server.url at the production site; src/lib/native.ts bridges haptics,
share and splash with web no-ops; the layout persists ?variant= for shells). The app's reason to
exist beyond the browser: NOTIFICATIONS (timed and conditional), haptics, an icon on the home
screen, and store presence.

## 1. Notifications: the headline feature

Philosophy first, because it is a product decision, not a technical one: Excogni does not nag.
The anti-gamification stance extends to notifications or the app contradicts itself. Concretely:
- Default state: ONE quiet daily reminder, off until the user turns it on.
- Hard caps: at most 1 scheduled reminder + 1 conditional notification per day, quiet hours
  (22:00 to 09:00 local, user adjustable) always respected, global mute switch in Settings.
- Honest copy only: "8 retention cards are due" (a fact), never "You're about to lose your
  streak!" (a guilt lever). No streak-threat notifications, ever: rest is part of a real routine.
- Every notification deep-links to the exact place it talks about.

### 1a. Timed reminders (local, no server, Phase 1)
@capacitor/local-notifications (add dependency), scheduled on-device:
- Daily practice reminder at a user-chosen time ("Time to practice" plus, if known from the last
  sync, "N cards due"). Rescheduled on every app open with fresh data.
- Implementation: a small native.ts extension (scheduleDailyReminder, cancelReminders) with web
  no-ops; a Settings block (time picker, on/off) stored in user_settings so web and app agree.
- Works fully offline; survives reboot (plugin handles boot rescheduling on Android).

### 1b. Conditional notifications, local flavor (Phase 1, clever and serverless)
On every app open/close, the shell fetches a tiny "what's ahead" payload from an endpoint
(GET /api/notify/plan) and schedules LOCAL notifications for future conditions already knowable:
- "Your retention cards come due tomorrow morning: 6 of them" scheduled for tomorrow 09:30.
- "Your reaction-time sample is getting stale (14 days)" scheduled 3 days ahead if applicable.
This gives conditional behavior with zero push infrastructure, at the cost of staleness if the
user does not open the app for long stretches. That trade-off is acceptable for Phase 1.

### 1c. Conditional notifications, push flavor (FCM, Phase 2)
For conditions only the server can see in time (weekly summary ready, percentile unlocked,
"your profile completed calibration"), real push:
- Client: @capacitor/push-notifications (dep already present), register on login, send token to
  POST /api/notify/register.
- Schema (migration): device_tokens (id, user_id FK cascade, platform, token unique, created_at,
  last_seen_at) and notification_prefs folded into user_settings (reminder_time, reminder_enabled,
  conditional_enabled, quiet_start, quiet_end, push_enabled).
- Server scheduler: a node-cron loop inside the existing app container (no new service), running
  every 15 minutes: evaluate per-user conditions (due cards crossing a threshold, weekly summary
  on Sunday evening local, staleness), respect quiet hours in the USER's timezone (practice_
  sessions.timezone already exists), respect caps, write a notification_log row (dedupe: never the
  same condition twice within its cooldown), send via FCM HTTP v1 with a service-account key in
  env. Log sends with log.info for the Health panel.
- HONESTY FLAG for the privacy posture: FCM is a Google service; the no-third-party claim gains an
  asterisk. Mitigate: send data-free pushes where possible (collapse to "tap to see"), document it
  in the privacy policy, and note UnifiedPush as the self-hosters' alternative in the open-source
  docs (roadmap item, not Phase 2).

## 2. Auth: magic links must open the app

Cookie-based magic links are the auth model; the link must land inside the shell's WebView or the
session lands in the browser instead of the app.
- Android App Links: host .well-known/assetlinks.json on the canonical domain (Caddy serves it;
  it lists the app's signing-cert SHA-256), intent-filter for https://xcgni.com/auth/magic* in
  AndroidManifest, autoVerify=true.
- The WebView shares the app's cookie jar, so once /auth/magic is opened IN the app, the session
  cookie persists app-side. Test the full loop on a real device: request link in app, open mail
  app, tap link, land logged-in back in Excogni.
- Fallback path for when link opens in browser anyway (user mail-app settings vary): the magic
  page shows "Open in the Excogni app" button (intent:// URL) when the UA is Android and the app
  is not the requester.

## 3. Native polish (the difference between "webview" and "app")

- Status bar and splash colors follow the chosen SCHEME, not hardcoded ink: small JS bridge call
  on scheme change (StatusBar.setBackgroundColor with the scheme's ink). make-variant.mjs already
  parameterizes this per variant at build time; runtime switch handles in-app changes.
- Android back button: Capacitor's backButton event -> history.back() when the WebView can go
  back, minimize (not exit) at the root. Without this the app dies on first back press.
- Keep-screen-on DURING reaction trials only (@capacitor-community/keep-awake): a dimming screen
  mid-trial ruins the measurement. Enable at trial start, disable at result.
- Offline state: remote-URL shells show an ugly error page with no network. Ship a minimal local
  error page (Capacitor errorPath) with the radar mark and a Retry button.
- Safe areas: viewport-fit=cover plus env(safe-area-inset-*) padding on the sticky header and
  bottom actions; test on a notched device.
- Haptics: already bridged (tapFeedback); extend to reaction cue and level-up moments, subtle.
- Text zoom: lock WebView textZoom to 100 so Android font-size settings do not break layouts
  (accessibility note: the app's own font-size setting remains the supported path).

## 4. Play Store production checklist

- Signing: generate an upload keystore, enroll in Play App Signing, BACK UP the keystore + store
  passwords in the password manager (losing it = losing the app identity).
- Target SDK: whatever Play currently mandates for new apps (verify at submission; it ratchets
  yearly). Set minSdk 26 (covers Capacitor 6 and ~97% of devices).
- Data Safety form: filled honestly and it will read WELL given the posture: email (optional,
  auth), self-reported context (optional), performance data; no ads, no third-party sharing
  (Google Play Services push delivery disclosed once FCM ships), deletion available in-app.
- Privacy policy URL: xcgni.com/privacy must exist and mention the app + FCM when applicable.
- Listing assets: icon (the radar mark on ink, 512px), feature graphic 1024x500 (radar sweep on
  graph paper), 4 to 6 phone screenshots (practice, radar, insights, calendar; Phosphor and Amber
  both photograph well), short description that leads with honest measurement.
- Testing: internal track first (you), then closed track (testers). NOTE: personal dev
  accounts created after Nov 2023 must run a closed test with 12+ testers for 14 days before
  production access; if the account is new, start this clock EARLY, in parallel with Phase 2.
- Content rating questionnaire, app category Education or Health & Fitness (Education fits the
  positioning better and has lighter policy scrutiny).

## 5. Build and release pipeline

- Version scheme: versionName = web VERSION (e.g. 0.60.0), versionCode = monotonic integer
  (date-based: 2026070201). The shell changes rarely; server-side changes ship instantly to it,
  which is the architecture's superpower: most releases need NO store submission.
- Build steps (document in android/README): node scripts/make-variant.mjs serious (regenerates
  capacitor.config.ts with the canonical domain), npm run build if shell assets changed, npx cap
  sync android, open in Android Studio or ./gradlew bundleRelease, upload .aab to Play.
- When a store release IS needed: only for native changes (plugins, manifest, icons). Everything
  else deploys with the VPS as today.
- CI later (GitHub Actions with a secured keystore) once open-sourced; manual builds are fine at
  this scale and keep the keystore off third-party infra.

## 6. Phases, acceptance criteria, effort

- PHASE 1 (shell + timed local): canonical domain fixed everywhere, App Links + magic-link loop
  verified on device, back button + offline page + safe areas + keep-awake, daily reminder with
  Settings UI, local conditional scheduling from /api/notify/plan. DONE WHEN: full journey
  (install -> login via email -> practice -> reminder fires next day -> tap lands in practice)
  works on your phone and a tester's. Effort: 2 to 4 focused sessions.
- PHASE 2 (push + scheduler): device_tokens migration, FCM project + service account, register
  endpoint, cron evaluator with caps/quiet-hours/dedupe log, weekly-summary + calibration-complete
  + due-threshold conditions, Health panel counters. DONE WHEN: a forced condition produces
  exactly one correctly-timed push that deep-links right, and the log proves dedupe. Effort: 3 to
  5 sessions.
- PHASE 3 (store): assets, forms, privacy page update, internal -> closed track (12-tester clock
  if applicable), crash-free week on closed track. DONE WHEN: production listing live.
- PHASE 4 (post-launch): monitor (Play vitals + server logs), then the deferred niceties:
  UnifiedPush for self-hosters, iOS shell (config already scaffolds it; needs a Mac + Apple
  account), widget with due-card count.

## 7. Risks, named honestly

- Domain mismatch: RESOLVED, canonical is xcgni.com across the shell tooling (keep the 301).
- FCM vs privacy posture: disclosed trade-off, mitigations above; do not pretend it is nothing.
- WebView input latency for reaction time differs from Chrome desktop: the per-device calibration
  already absorbs this by design; verify the floor looks sane on 2 to 3 real devices.
- Play review of remote-URL apps: policy-compliant when it is your own service with native value
  (notifications, haptics, deep links provide exactly that); the listing copy should present it
  as the product, never "a website wrapper".
- The 12-tester/14-day rule can gate the launch date: check account status NOW and start the
  clock in parallel, not after Phase 2.
